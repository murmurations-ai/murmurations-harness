### 4.1 Init UX overhaul (single PR)

**Goal:** operator pastes API keys once, never edits `.env` by hand.

Changes to `packages/cli/src/init.ts`:

- **Interactive secret capture.** After the LLM provider question, prompt for the relevant `<PROVIDER>_API_KEY` with **echo-off** (write directly to the generated `.env`, never to stdout). Keys accepted:
  - `gemini` â†’ `GEMINI_API_KEY`
  - `anthropic` â†’ `ANTHROPIC_API_KEY`
  - `openai` â†’ `OPENAI_API_KEY`
  - `ollama` â†’ no key needed (skip, note locally-hosted)
- **GitHub token capture.** If `collaboration: github`, prompt for `GITHUB_TOKEN` (also echo-off). Paste or ENTER to skip (init warns "you'll need to add it before any harness write").
- **Offer to validate keys live.** After capture, offer to make a tiny live call to the LLM provider (1-token completion) and `GET /user` to GitHub. Skippable. If both pass, `init` prints "âś“ Credentials validated."
- **Generate `.env.example` alongside `.env`.** Operator commits `.env.example` (for teammates/CI), `.gitignore` covers `.env`.
- **Pre-init check for existing state.** If `murmuration/` already exists, detect whether it's a current, legacy (flat-file), or partial state and offer to:
  - abort,
  - overlay missing pieces (ADR-0026-compliant upgrade path â€” the migration we just did for EP),
  - or overwrite with a fresh scaffold.
- **Post-init workflow teaser.** Replace the current "Next steps" block with a hero command:

  ```
  âś“ Murmuration initialized. Validate and run:

     murmuration doctor                          # verify setup
     murmuration group-wake --group <first>     # your first meeting
  ```

- **Fix three existing minor gaps:** emit `tools.mcp:` and `plugins:` keys in generated `role.md` (currently missing; present in the default-agent template but not in init-generated agents). Emit a short intro comment at the top of `role.md` explaining what each field means (link to ADR-0016 reference).

**Non-goals for this PR:**

- Re-architecting the interview flow
- Changing the generated directory layout (ADR-0026 stands)
- Adding new governance model plugins

### 4.3 `murmuration doctor` command (separate PR)

**Goal:** one command diagnoses whether a murmuration will run, and tells the operator exactly what to fix if not.

New subcommand: `murmuration doctor [--root <path>] [--fix] [--json]`.

**The rubric â€” what a well-configured murmuration looks like:**

Each check has `severity: error | warning | info`, `category`, `remediation` text, and optional `--fix` auto-remediation. `error` fails the exit code; `warning` prints but returns 0; `info` is advisory.

**Category 1 â€” Layout (ADR-0026)**

- [ ] `murmuration/` exists
- [ ] `murmuration/harness.yaml` exists and parses as YAML
- [ ] `murmuration/soul.md` exists and non-empty
- [ ] `murmuration/default-agent/{soul,role}.md` exist
- [ ] `agents/` exists
- [ ] At least one `agents/<slug>/` directory
- [ ] `governance/groups/` exists if any agent declares `group_memberships:`
- [ ] `.gitignore` exists and covers `.env`
- [ ] `.env` has `0600` permissions (if exists)
- [ ] Legacy paths not mixed with canonical (e.g., no `governance/circles/` alongside `governance/groups/`; suggest `doctor --fix` to rename)

**Category 2 â€” Schema (ADR-0016)**
For each `agents/<slug>/role.md`:

- [ ] Parses as YAML
- [ ] `agent_id` is a string (not number) and matches the directory slug
- [ ] `name` is non-empty string
- [ ] `llm.provider` is one of `gemini | anthropic | openai | ollama`
- [ ] `model_tier` is `fast | balanced | deep`
- [ ] `group_memberships` is an array of strings (if present)
- [ ] `soul_file` resolves to an existing file (defaults to `soul.md`)
- [ ] `wake_schedule.cron` is a valid cron string (if present)
- [ ] `budget.max_cost_micros` > 0 and realistic for the agent's workload

For each `governance/groups/<id>.md`:

- [ ] Has a parseable `## Members` section with â‰Ą1 member
- [ ] Has a `facilitator:` line
- [ ] Every member refers to an existing `agents/<slug>/` directory
- [ ] `facilitator` is one of the members

For `murmuration/harness.yaml`:

- [ ] Has `llm.provider` matching one of the valid providers
- [ ] `collaboration.provider` is `github | local`
- [ ] If `collaboration.provider: github`, `collaboration.repo` is `owner/name` shape

**Category 3 â€” Secrets**

- [ ] `.env` exists (warning if absent)
- [ ] `<PROVIDER>_API_KEY` present and non-empty for the configured LLM provider
- [ ] `GITHUB_TOKEN` present if `collaboration.provider: github`
- [ ] `.env` is ignored by git (check `.gitignore` + `git ls-files`)
- [ ] `.env` has `0600` permissions (warn if more permissive)

**Category 4 â€” Live validation** (opt-in, `--live` flag, skipped by default)

- [ ] LLM provider responds to a 1-token completion call
- [ ] GitHub token is valid (`GET /user` returns 200)
- [ ] GitHub token has `repo` scope on the configured repo

**Category 5 â€” Governance wiring**

- [ ] If `governance.plugin` is set in harness.yaml, the plugin module resolves
- [ ] If no governance plugin, warn that `group-wake` meetings will run with no state machine

**Category 6 â€” Drift / best-practice** (info-level)

- [ ] Any `role.md` still has a `# TODO:` marker
- [ ] Any agent has empty `github.write_scopes.*` arrays but `wake_schedule` set (likely un-productive wakes)
- [ ] Any group has more than 8 members (S3 best-practice: split)

## 5. Decision: where hello-circle lives

**Recommendation: bundle the example inside the CLI package, invoked via `murmuration init --example hello`. Do NOT ship it as a separate repo.**

Three options considered:

| Option                                                                                        | UX                                                                           | Drift risk                                      | Implementation cost           |
| --------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- | ----------------------------------------------- | ----------------------------- |
| A. `packages/examples/hello-circle/` in the monorepo (operators clone the whole harness repo) | Poor â€” requires monorepo knowledge, pnpm install, `cd packages/examples/...` | Low â€” CI tests it                               | Low                           |
| B. Separate repo `murmurations-ai/hello-circle` (operators `git clone` that)                  | OK â€” two commands: `git clone` then `cp .env.example .env`                   | High â€” example drifts from harness API          | Medium (new repo to maintain) |
| C. **Bundled in CLI: `murmuration init --example hello`**                                     | **Best â€” one command, no git clone, no context-switch**                      | **Low â€” lives in the monorepo next to init.ts** | **Low â€” copy template files** |

Why C wins:

- **Zero cognitive tax.** A tester who just installed `@murmurations-ai/cli` stays in that mental model. They don't have to know the harness is a monorepo. They don't navigate to a GitHub org page to find the example.
- **Single source of truth.** The hello-circle ships in the same git commit as the CLI that runs it. CI tests the init-to-group-wake round-trip against the same example code.
- **Trivially extensible.** Later we add `--example research-trio`, `--example code-review-crew`, etc. The pattern is the same flag, zero additional infrastructure.
- **Offline-friendly.** No GitHub fetch; templates are bundled in the npm package's `dist/examples/`.

What to do as a secondary path:

- **Also** ship the hello-circle as a `murmurations-ai/hello-circle-example` repo so it's browsable without installing the CLI. The repo's contents are `git clone`'d directly into the monorepo via a pre-commit sync script (or the monorepo IS the canonical; the standalone repo is generated from it). That way marketing-side onboarding ("look at this example on GitHub") still works.
- The two paths converge: `murmuration init --example hello` and `git clone murmurations-ai/hello-circle-example` produce the same directory tree.

## 8. Risks

**Risk: `doctor` becomes a second source of truth that drifts from the Zod schema.**
Mitigation: `doctor` imports `roleFrontmatterSchema` from core and runs Zod itself for category 2 checks. No re-implementation. The rubric lives in tests + docs, not in a parallel validator.

**Risk: interactive secret capture breaks in CI or non-TTY environments.**
Mitigation: detect non-TTY on stdin; fall back to reading `INIT_GEMINI_API_KEY` from environment or writing placeholders (current behavior). Print "not a TTY â€” skipping interactive secret capture" explicitly.

**Risk: `--example hello` bundle bloats the npm package.**
Mitigation: the whole hello-circle is ~20KB of markdown + a tiny `harness.yaml`. Negligible.

**Risk: "live validation" burns tester's free-tier API quota uselessly.**
Mitigation: opt-in (`--live` or `murmuration doctor --live`); init prompts "Validate credentials now? (y/N)" with default N. One 1-token completion is â‰¤$0.0001 anyway.

**Risk: the big-bang PR is too big to review.**
Mitigation: the "big bang" is really 5 sequenced milestones, each a single reviewable PR. Nori's "one init UX overhaul PR" direction is honored by bundling Â§4.1 into a single PR; the other milestones ship alongside but independently.

**Risk: testers hit a real harness bug outside v0.5.0's scope and bounce.**
Mitigation: `doctor` doesn't prevent all bugs; it prevents configuration bugs. Known runtime bugs (cross-repo writes, etc.) get clear limitation notices in `GETTING-STARTED.md` and `doctor` info-level warnings.

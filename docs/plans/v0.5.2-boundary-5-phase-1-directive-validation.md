# Implementation Plan v0.5.2 — Boundary 5 Phase 1: Directive Validation

- **Status:** Draft — pending Source approval
- **Date:** 2026-04-30
- **Tracks:** [#239](https://github.com/murmurations-ai/murmurations-harness/issues/239); Proposal 07 § Boundary 5
- **Implements:** Layer 1 of the Boundary 5 fix-stack ("require structured action evidence for directive signals")
- **Defers:** Layer 2 (API-level `tool_choice: required`) and Layer 3 (full narrative ↔ tool-call diff) to follow-up plans

## Problem

`packages/core/src/execution/index.ts:557` — current `validateWake()` counts an action item as "addressed" if any of three conditions hold:

```ts
const referenced =
  result.actions.some((a) => a.issueNumber === issueNum) ||
  actionReceipts.some((r) => r.action.issueNumber === issueNum) ||
  result.wakeSummary.includes(`#${String(issueNum)}`); // ← THIS LINE
```

The third clause is the bug. The agent's narrative output (the `wakeSummary`) is freeform LLM text; mere mention of `#592` causes the wake to count #592 as "addressed" even when no real action was executed. Verified empirically with architecture-agent (xeeban/emergent-praxis 2026-04-30): three wakes produced narrative claiming "I have posted my CONSENT on issue #592" with `tool_calls: 0` and zero comments actually landing on GitHub. `actionItemsAddressed: 5` reported despite no real GitHub mutations on those items. False governance records compound across wakes.

## Goal

Tighten `validateWake()` so that **a directive signal is only counted as addressed when there is structured evidence of action**: a `WakeAction` targeting the issue with a successful `WakeActionReceipt`, OR a governance event explicitly referencing it (the legitimate "I can't act, here's why" path). Narrative mention in `wakeSummary` no longer counts.

When directives in the wake's signal bundle are not addressed by structured evidence, the validation result includes a `directivesUnaddressed` list. The daemon translates the presence of this list into a non-productive outcome and surfaces a clear log event.

## Non-goals (deferred)

- **API-level `tool_choice: "required"` (Layer 2).** A separate follow-up plan; touches all four LLM provider adapters.
- **Narrative ↔ tool-call diff (Layer 3).** The full Boundary 5 fix per Proposal 07 § Boundary 5. Requires extracting action verbs from freeform narrative and reconciling against tool-call logs.
- **Model-policy update for EP.** A separate operator-level decision (set `llm.provider: anthropic` for engineering-circle agents); orthogonal to runtime fix and lives in EP role.md, not the harness.
- **Closing or commenting on architecture-agent's hallucinated consents on EP #571 / #554.** Operator cleanup; not harness work.

## Design

### 1. Type changes

`packages/core/src/execution/index.ts` — extend `WakeValidationResult`:

```ts
export interface UnaddressedDirective {
  /** Issue number of the unaddressed directive. */
  readonly issueNumber: number;
  /** Why it wasn't counted as addressed. Always one of these literal codes. */
  readonly reason:
    | "no-structured-action" // no WakeAction targeting this issue
    | "no-successful-receipt" // had an action but receipt failed
    | "narrative-only-claim" // wakeSummary mentions the issue but no action
    | "governance-event-missing"; // had no tension/proposal referencing it either
}

export interface WakeValidationResult {
  readonly productive: boolean;
  readonly artifactCount: number;
  readonly actionItemsAddressed: number;
  readonly actionItemsAssigned: number;
  /**
   * Directives present in the wake's signals that were NOT addressed by
   * structured evidence (action + receipt) or governance event. Empty
   * array when all directives were addressed or no directives were present.
   * Narrative-only mention of a directive in wakeSummary does NOT count
   * as addressing it (Boundary 5).
   */
  readonly directivesUnaddressed: readonly UnaddressedDirective[];
  readonly reason?: string;
}
```

### 2. Validation logic changes

`validateWake()` gets a new third argument: the full `signals` (currently it only receives `actionItems`, but directives can be in either `signals` or `actionItems`):

```ts
export const validateWake = (
  context: { actionItems: readonly Signal[]; signals: readonly Signal[] },
  result: { ... },
  actionReceipts: readonly WakeActionReceipt[],
): WakeValidationResult => {
  // ... existing artifactCount, actionItemsAddressed logic, but with the
  //     wakeSummary.includes() clause REMOVED for action-item matching ...

  // NEW: directive validation
  const directivesUnaddressed: UnaddressedDirective[] = [];
  const allSignals = [...context.signals, ...context.actionItems];
  for (const sig of allSignals) {
    if (sig.kind !== "github-issue") continue;
    const labels = (sig as unknown as { labels: readonly string[] }).labels;
    if (!labels.includes("source-directive")) continue;
    const issueNum = (sig as unknown as { number: number }).number;

    const hasAction = result.actions.some((a) => a.issueNumber === issueNum);
    const hasReceipt = actionReceipts.some(
      (r) => r.action.issueNumber === issueNum && r.success,
    );
    const hasGovEvent = result.governanceEvents.some((g) =>
      String(g.payload).includes(`#${String(issueNum)}`)
        || String(g.payload).includes(String(issueNum)),
    );

    if (hasReceipt) continue;                                  // addressed
    if (hasGovEvent) continue;                                 // explicitly tensioned
    if (hasAction && !hasReceipt) {
      directivesUnaddressed.push({ issueNumber: issueNum, reason: "no-successful-receipt" });
    } else if (result.wakeSummary.includes(`#${String(issueNum)}`)) {
      directivesUnaddressed.push({ issueNumber: issueNum, reason: "narrative-only-claim" });
    } else {
      directivesUnaddressed.push({ issueNumber: issueNum, reason: "no-structured-action" });
    }
  }

  // ... existing productive/reason logic, but with directivesUnaddressed
  //     forcing productive=false when non-empty ...
};
```

### 3. Daemon handling

`packages/core/src/daemon/index.ts` — when validation reports `directivesUnaddressed.length > 0`, the daemon:

1. Logs `daemon.wake.directives.unaddressed` at WARN with the list of issue numbers and reasons.
2. Marks the wake as non-productive in the agent state (consistent with current `productive: false` behavior — increments the idle-wake counter, contributes to circuit-breaker count).
3. Does NOT change the `outcome.kind` from `"completed"` to `"failed"` in this phase. Rationale: the LLM completed; the LLM's output was just unproductive. We surface the discrepancy without blowing up wake bookkeeping in v0.5.2. A future operator-configurable strict mode (Layer 2) can fail the wake outright.

This is a deliberate "warn, don't fail" posture for v0.5.2 to keep blast radius small. Operators upgrading to v0.5.2 see new log events and unproductive marks; they don't see existing wake-success-rate metrics flip to failures overnight.

### 4. Wake-summary wake's-eye reporting

`packages/core/src/runner/index.ts` — when `directivesUnaddressed.length > 0`, append to the wake summary:

```
  directives_unaddressed: 3 (#592 narrative-only, #571 narrative-only, #554 narrative-only)
```

This makes the discrepancy visible in run digests so operators reading them see immediately that the agent's narrative claims weren't backed by action.

### 5. Self-reflection demotion (light version)

`packages/core/src/runner/index.ts` — when `directivesUnaddressed.length > 0` and the LLM self-reported `effectiveness: high`, downgrade to `effectiveness: low` in the digest. The digest annotates the override:

```
EFFECTIVENESS: low (downgraded from agent-reported 'high' due to 3 unaddressed directives)
```

This is the tiniest possible Layer 3 element — does not require parsing the narrative, just looks at the validation result. The agent's grade is overridden when their work doesn't back up their grade.

## Risks

1. **False positive on legitimately-unaddressable directives.** If an agent genuinely cannot act on a directive (missing tool, repo unreachable), they should file a tension referencing the issue number. The validator counts that as addressed via the governance-event path. Risk mitigated.
2. **Breaking tests in downstream consumers.** `validateWake`'s signature changes. The audit list in Step 1 includes "find all call sites" — we expect these to be in daemon, runner, and tests.
3. **Operator confusion at first wave of unproductive wakes.** Mitigated by clear log event name and digest annotation. Documented in OBSERVABILITY.md.

## Open questions for Source

1. **Effectiveness downgrade — keep or split?** The `effectiveness` downgrade in the digest is a tiny Layer 3 element. Including it makes the failure surface visible to anyone reading the digest, not just operators reading daemon logs. I recommend keeping it. If you disagree, we ship Steps 1–4 only.
2. **Should narrative-only matching also be removed for non-directive action items?** Currently the `wakeSummary.includes("#N")` clause matches all action items, not just directives. Removing it for directives only (this plan) is conservative. Removing it for all action items is more correct (same hallucination class) but increases blast radius for v0.5.2. **Recommendation:** remove for directives only in v0.5.2; remove for all action items in v0.5.3 once we have empirical data on whether the tightened directive validation is producing useful signal without too many false positives.
3. **Branch + PR naming.** Suggest `feat/b5-phase-1-directive-validation` branch, opened as a PR against `main`, merged after the proposal-07 consent round closes.
